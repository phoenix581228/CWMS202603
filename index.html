1 Google OAuth + 裝置綁定 + Nginx 反向代理

🎯 要解決的痛點

社工外勤訪視時，需登入 CWMS 內網紀錄個案資料。目前需透過 VPN 或回辦公室操作，造成：作業延遲、VPN 不穩報修多、行動裝置體驗差。這是 112 項需求中「介面與行動裝置體驗」類約 15 項的根本成因之一。

技術架構分層

📱 社工行動裝置
Chrome / Safari
▼ HTTPS (443)
Layer 1：Nginx 反向代理（DMZ）
✅ SSL/TLS 終結（Let's Encrypt）
✅ 請求過濾 / Rate Limiting / WAF 規則
✅ 靜態資源快取
✅ Header 注入 X-Forwarded-* / X-Device-ID
▼ 內部轉發 (8080)
Layer 2：認證中介層（Auth Gateway）
🔐 Google OAuth 2.0 驗證 → 取得 @tzuchi.org.tw 身份
📲 裝置指紋綁定（Device Fingerprint + UUID）
🎫 簽發 JWT Token（含 user_id + device_id + 角色）
⏱️ Token 有效期管理（access: 30min / refresh: 7days）
▼ 帶 JWT 轉發
Layer 3：CWMS 內網應用
✅ 驗證 JWT 簽名與裝置 ID 一致性
✅ 現有登入邏輯不變（JWT user_id 對應 CWMS 帳號）
✅ 原有內網直連仍然保留

🔗 裝置綁定機制設計

項目說明
裝置指紋 首次登入時收集：瀏覽器 User-Agent + 螢幕解析度 + 語言 + 時區 → 產生 SHA-256 hash 作為 Device Fingerprint
裝置 UUID 透過 Service Worker 於瀏覽器端生成唯一 UUID，存入 IndexedDB（比 localStorage 更持久）
綁定流程 首次 OAuth 成功 → 管理員審核裝置 → 寫入白名單 → 後續登入自動比對 Device Fingerprint + UUID
裝置上限 每位社工最多綁定 3 台裝置（手機 + 平板 + 電腦），超出需管理員審核解綁
異常偵測 裝置指紋不符 → 觸發二次驗證（OTP 簡訊）；連續 5 次失敗 → 帳號鎖定通知管理員

⚙️ Nginx 反向代理關鍵配置

配置項建議值用途
SSL Certificate Let's Encrypt (自動續期) HTTPS 加密傳輸
proxy_pass http://cwms-internal:8080 轉發至內網 CWMS
rate_limit 30 req/min per IP 防止暴力攻擊
client_max_body_size 20M 附件上傳限制
auth_request /auth/verify-token 每個請求先過認證中介層
proxy_set_header X-Device-ID, X-Real-IP 傳遞裝置與來源資訊
geo $blocked 境外 IP 段封鎖 僅允許台灣 IP 存取

🛡️ 安全防護層設計

✅ 身份安全

限定 @tzuchi.org.tw 組織帳號登入，非組織帳號直接拒絕

✅ 裝置安全

裝置綁定 + 指紋比對，遺失裝置可遠端撤銷

✅ 傳輸安全

全程 HTTPS + HSTS，Nginx 為唯一對外入口

✅ 稽核追蹤

完整 access log（who + when + device + action），符合個資法要求

✅ 優勢

  • 零改動 CWMS 程式碼：代理層獨立於 CWMS，不影響現有功能
  • 社工可在外用手機直接操作 CWMS，免 VPN
  • 緩解約 15 項「行動裝置 / 連線」相關優化需求
  • Google OAuth 與慈濟既有 Google Workspace 直接整合
  • Nginx 開源免授權費，維運成本低

⚠️ 風險與挑戰

  • CWMS 內網開一個對外入口，需嚴格安全審查
  • 裝置綁定管理（離職/換機）需建立管理流程
  • 需確認資安政策允許透過代理存取內網系統
  • 網路組需配合開放 DMZ 規則
  • 高負載時 Nginx 需做 HA (Active-Standby)

📅 實施階段規劃

建立 DMZ 區 Nginx 伺服器 → 設定 SSL 憑證 → Google Cloud Console 建立 OAuth Client → 基礎反向代理測試

Phase 2（2 週）— 認證中介層開發

開發 Auth Gateway 微服務（Node.js / Python）→ JWT 簽發與驗證 → 裝置指紋收集與綁定邏輯 → 裝置管理後台

Phase 3（1 週）— 整合測試

CWMS 端 JWT 驗證整合 → 端對端測試（內網+外網）→ 安全滲透測試 → 效能壓測

Phase 4（2 週）— 試行推廣

選定 1-2 區社工試用 → 收集回饋 → 調整 → 全區推廣

預估工時：約 7 週（1 位後端 + 1 位網路/資安）

3 CWMS 志工名單 + SSO 統一認證

🎯 核心目的

目前 CWMS 的訪視志工名單管理分散，且 CWMS / CVRS / AI 寶各有獨立帳號體系。本方案建立統一身份認證（SSO），從 CWMS 拉出志工名單作為唯一人員清冊來源。考量志工年齡層偏高，初期以「手機簡訊 OTP」為主要登入方式（最易被接受），同時保留 Google OAuth 管道。採雙軌併行策略，待日後新世代志工接軌，再逐步以 Google OAuth 為主。

📋 CWMS 志工名單整合方案

項目設計說明
資料來源 以 CWMS 資料庫中的訪視志工名冊為唯一真實來源（Single Source of Truth），包含：志工編號、姓名、手機號、所屬組隊、服務區域、狀態
同步機制 SSO 服務每日排程從 CWMS DB 同步志工名單 → 異動時即時觸發 webhook → AI 寶同步更新
帳號自動建立 志工首次使用 AI 寶時，手機號比對 CWMS 名單 → 自動建立 SSO 帳號 → 免人工申請
離線處理 志工離隊 → CWMS 名單異動 → SSO 自動停用帳號 → AI 寶/CWMS 同步吊銷存取權
角色對應 CWMS 職務 → SSO Role 映射：社工=social_worker（完整權限）、志工=volunteer（訪視相關權限）、組長=team_lead（＋團隊管理）

📲 手機號 OTP 實作方案比較

方案 服務商 費用 優點 限制
推薦 Twilio Verify ~NT$1.5/則 全球通用、SDK 完善、自動防欺詐 國際服務商

🔄 雙軌併行 → 世代轉換路線圖

考量現有志工年齡層偏高，手機簡訊 OTP 為最低學習門檻的登入方式。系統設計為雙軌並存，隨新世代志工加入逐步提高 Google OAuth 佔比：

階段 時間軸 簡訊 OTP 佔比 Google OAuth 佔比 策略說明
Phase 1 上線初期 90% 10% 所有志工預設簡訊登入，僅有組織信箱的社工/幹部使用 OAuth
Phase 2 1-2 年 60% 40% 新加入志工引導使用 Google 帳號登入；既有志工維持簡訊不強迫切換
Phase 3 3-5 年 30% 70% Google OAuth 為主要推薦方式，簡訊保留為備用，節省簡訊費用
長期 5 年+ 備用 主力 新世代全面 OAuth，簡訊 OTP 僅保留給無 Google 帳號的特殊情境
💡 關鍵原則：不強迫現有志工改變習慣 → 雙軌永遠並存 → 透過自然世代交替逐步轉換 → 系統設計上兩種方式權重相同，僅推薦順序隨時間調整

✅ 優勢

  • 一次建設、全線受益：方案一和方案二都依賴此 SSO 層
  • 簡訊登入零學習成本：志工只需會收簡訊，無需記帳密
  • 雙軌併行尊重現有志工習慣，不造成抗拒
  • 帳號生命週期自動化（入隊/離隊自動處理）
  • 統一稽核日誌，資安合規
  • 世代交替自然轉換，長期降低簡訊費用

⚠️ 風險與挑戰

  • CWMS 志工名單資料品質（手機號缺漏/錯誤）需先清理
  • 初期簡訊為主，費用隨使用量成長（可搭配免費額度控制）
  • 需製作簡易圖示教學，協助年長志工理解「收簡訊→輸入驗證碼」流程
  • SSO 為核心基礎設施，需高可用性設計
  • 需與 CWMS 資料庫建立 API 或 DB view 介接

📅 實施階段規劃
